---
title: "Using Spring and Spring Boot with Cucumber JVM"
description: "Wire Cucumber step definitions into a Spring ApplicationContext with autowiring, scenario scope, mock beans, and transactional test support."
---

The `cucumber-spring` module integrates Cucumber JVM with Spring's `TestContextManager` framework. It gives each test run a single shared `ApplicationContext`, scopes all step definition beans to the current scenario so state never leaks between scenarios, and supports the full range of Spring test features including `@MockBean`, `@Transactional`, and `@DirtiesContext`.

## Adding the dependency

Add `cucumber-spring` to your Maven `pom.xml`. Use the `cucumber-bom` for dependency management so you do not need to pin a version manually.

```xml pom.xml
<dependencyManagement>
  <dependencies>
    <dependency>
      <groupId>io.cucumber</groupId>
      <artifactId>cucumber-bom</artifactId>
      <version>${cucumber.version}</version>
      <type>pom</type>
      <scope>import</scope>
    </dependency>
  </dependencies>
</dependencyManagement>

<dependencies>
  <dependency>
    <groupId>io.cucumber</groupId>
    <artifactId>cucumber-spring</artifactId>
    <scope>test</scope>
  </dependency>
</dependencies>
```

## Configuring the application context

Cucumber needs to know which Spring configuration to use. Create a dedicated configuration class on your glue path and annotate it with `@CucumberContextConfiguration` plus one of the standard Spring test annotations.

<Tabs>
  <Tab title="Spring Boot">
    ```java src/test/java/com/example/app/CucumberSpringConfiguration.java
    package com.example.app;

    import io.cucumber.spring.CucumberContextConfiguration;
    import org.springframework.boot.test.context.SpringBootTest;

    @CucumberContextConfiguration
    @SpringBootTest(classes = TestConfig.class)
    public class CucumberSpringConfiguration {

    }
    ```
  </Tab>
  <Tab title="Spring Framework">
    ```java src/test/java/com/example/app/CucumberSpringConfiguration.java
    package com.example.app;

    import io.cucumber.spring.CucumberContextConfiguration;
    import org.springframework.test.context.ContextConfiguration;

    @CucumberContextConfiguration
    @ContextConfiguration(classes = TestConfig.class)
    public class CucumberSpringConfiguration {

    }
    ```
  </Tab>
</Tabs>

<Warning>
The class annotated with `@CucumberContextConfiguration` is **not** a step definition class. Do not put `@Given`, `@When`, or `@Then` methods in it. It is processed by Spring's test execution listeners — not initialized as a regular Spring bean — so features like AspectJ that depend on bean initialization will not work on this class.
</Warning>

## Autowiring beans into step definitions

Step definition classes are initialized by Spring as regular beans. You can inject dependencies using field injection or constructor injection.

<CodeGroup>

```java Field injection
package com.example.app;

import io.cucumber.java.en.Given;
import org.springframework.beans.factory.annotation.Autowired;

public class MyStepDefinitions {

    @Autowired
    private MyService myService;

    @Given("feed back is requested from my service")
    public void feed_back_is_requested() {
        myService.requestFeedBack();
    }
}
```

```java Constructor injection
package com.example.app;

import io.cucumber.java.en.Given;

public class MyStepDefinitions {

    private final MyService myService;

    public MyStepDefinitions(MyService myService) {
        this.myService = myService;
    }

    @Given("feed back is requested from my service")
    public void feed_back_is_requested() {
        myService.requestFeedBack();
    }
}
```

</CodeGroup>

## Sharing state between step definition classes

All glue code beans (step definitions, hooks) are bound to the `cucumber-glue` scope. This scope starts before each scenario and ends after it, so state is never shared across scenarios.

To share state between two step definition classes within the same scenario, create a dedicated component annotated with `@ScenarioScope` and autowire it into each class that needs it.

```java src/test/java/com/example/app/TestUserInformation.java
package com.example.app;

import io.cucumber.spring.ScenarioScope;
import org.springframework.stereotype.Component;

@Component
@ScenarioScope
public class TestUserInformation {

    private User testUser;

    public void setTestUser(User testUser) {
        this.testUser = testUser;
    }

    public User getTestUser() {
        return testUser;
    }
}
```

Both step definition classes autowire the same `TestUserInformation` instance for the duration of the scenario:

```java src/test/java/com/example/app/UserStepDefinitions.java
package com.example.app;

import io.cucumber.java.en.Given;
import org.springframework.beans.factory.annotation.Autowired;

public class UserStepDefinitions {

    @Autowired
    private UserService userService;

    @Autowired
    private TestUserInformation testUserInformation;

    @Given("there is a user")
    public void there_is_a_user() {
        User testUser = userService.createUser();
        testUserInformation.setTestUser(testUser);
    }
}
```

```java src/test/java/com/example/app/PurchaseStepDefinitions.java
package com.example.app;

import io.cucumber.java.en.When;
import org.springframework.beans.factory.annotation.Autowired;

public class PurchaseStepDefinitions {

    @Autowired
    private PurchaseService purchaseService;

    @Autowired
    private TestUserInformation testUserInformation;

    @When("the user makes a purchase")
    public void the_user_makes_a_purchase() {
        User user = testUserInformation.getTestUser();
        purchaseService.purchase(user, new Order());
    }
}
```

## Using mock beans

Declare mock beans on the `@CucumberContextConfiguration` class. They are then available for autowiring in all step definition classes.

```java src/test/java/com/example/app/CucumberSpringConfiguration.java
package com.example.app;

import io.cucumber.spring.CucumberContextConfiguration;
import org.springframework.boot.test.context.SpringBootTest;
import org.springframework.boot.test.mock.mockito.MockBean;

@CucumberContextConfiguration
@SpringBootTest(classes = TestConfig.class)
@MockBean(MyService.class)
public class CucumberSpringConfiguration {

}
```

```java src/test/java/com/example/app/MyStepDefinitions.java
package com.example.app;

import io.cucumber.java.en.Given;
import org.springframework.beans.factory.annotation.Autowired;

import static org.mockito.Mockito.mockingDetails;
import static org.springframework.test.util.AssertionErrors.assertTrue;

public class MyStepDefinitions {

    @Autowired
    private MyService myService;

    @Given("my service is a mock")
    public void my_service_is_a_mock() {
        assertTrue(mockingDetails(myService).isMock());
    }
}
```

## Dirtying the application context

Cucumber reuses the same `ApplicationContext` for all scenarios to avoid an expensive startup on every test. If a scenario modifies application state in a way that affects other tests, add `@DirtiesContext` to your configuration class. Spring will reload the context before the next test.

```java src/test/java/com/example/app/CucumberSpringConfiguration.java
package com.example.app;

import io.cucumber.spring.CucumberContextConfiguration;
import org.springframework.boot.test.context.SpringBootTest;
import org.springframework.test.annotation.DirtiesContext;

@CucumberContextConfiguration
@SpringBootTest(classes = TestConfig.class)
@DirtiesContext
public class CucumberSpringConfiguration {

}
```

<Warning>
Using `@DirtiesContext` together with parallel scenario execution produces undefined behaviour. Avoid combining them.
</Warning>

## Multiple application contexts

A single Cucumber execution supports only one `ApplicationContext`. If you need to test multiple application configurations, run Cucumber multiple times — once per configuration.

<Tabs>
  <Tab title="JUnit 4 / TestNG">
    ```java src/test/java/com/example/ApplicationOneTest.java
    package com.example;

    import io.cucumber.junit.Cucumber;
    import io.cucumber.junit.CucumberOptions;
    import org.junit.runner.RunWith;

    @RunWith(Cucumber.class)
    @CucumberOptions(
        glue = "com.example.application.one",
        features = "classpath:com/example/application/one"
    )
    public class ApplicationOneTest {

    }
    ```
  </Tab>
  <Tab title="JUnit 5 Platform Suite">
    ```java src/test/java/com/example/ApplicationOneTest.java
    package com.example;

    import org.junit.platform.suite.api.ConfigurationParameter;
    import org.junit.platform.suite.api.SelectPackages;
    import org.junit.platform.suite.api.Suite;

    import static io.cucumber.junit.platform.engine.Constants.GLUE_PROPERTY_NAME;

    @Suite
    @SelectPackages("com.example.application.one")
    @ConfigurationParameter(key = GLUE_PROPERTY_NAME, value = "com.example.application.one")
    public class ApplicationOneTest {

    }
    ```
  </Tab>
</Tabs>

Repeat the pattern for each application configuration you need to test.
